# PostgreSQL Database Design & Operations

## 🎯 Goal

Store uploaded files physically on the filesystem (`files/<user_id>/`) while maintaining PostgreSQL metadata and tags. Support CRUD operations for both files and tags, ensuring file deletion cleans up both database and filesystem.

## 🧱 Core Requirements

- `user_id` (Telegram user identifier)
- `file_path` (relative path, e.g. `123/myfile.pdf`)
- Multiple tags per file (generated by LLM or user)
- Physical file storage at `files/<user_id>/<filename>`

## 🗄️ Database Schema

### 1. `user`

| Field    | Type    | Constraints         |
|----------|---------|---------------------|
| id       | INTEGER | PRIMARY KEY         |
| username | TEXT    | NOT NULL DEFAULT '' |

### 2. `file`

| Field      | Type      | Constraints                       |
|------------|-----------|-----------------------------------|
| id         | SERIAL    | PRIMARY KEY                       |
| user_id    | INTEGER   | FK → user(id), NOT NULL           |
| file_path  | TEXT      | NOT NULL (relative, e.g. `123/f.pdf`) |
| created_at | TIMESTAMP | DEFAULT NOW()                     |

**CASCADE:** If user is deleted, files are deleted.

### 3. `tag`

| Field | Type   | Constraints       |
|-------|--------|-------------------|
| id    | SERIAL | PRIMARY KEY       |
| name  | TEXT   | UNIQUE, NOT NULL  |

### 4. `file_tag` (many-to-many)

| Field   | Type    | Constraints              |
|---------|---------|--------------------------|
| file_id | INTEGER | FK → file(id), NOT NULL  |
| tag_id  | INTEGER | FK → tag(id), NOT NULL   |

**PK:** `(file_id, tag_id)`. **CASCADE:** If file is deleted, file_tags are deleted.

### Indexes

- `idx_file_user_id` on `file(user_id)`
- `idx_file_tag_file_id` on `file_tag(file_id)`
- `idx_file_tag_tag_id` on `file_tag(tag_id)`

## 💾 File Storage

### Structure
```
files/
├── 123/
│   ├── report.pdf
│   └── notes.txt
└── 456/
    └── data.csv
```

- **Root:** `files/`
- **User subfolder:** `files/<user_id>/`
- **Full path:** `files/<user_id>/<filename>`
- **DB stores:** relative path `123/report.pdf`
- **Auto-create:** user folder is created on first upload

### Edge Cases

| Scenario                          | Handling                                  |
|-----------------------------------|-------------------------------------------|
| File with same name already exists | Raise 409 Conflict                        |
| File deletion fails (disk error)  | Raise 500, do NOT delete DB record        |
| User folder missing               | Auto-create on first upload               |
| DB insert fails after disk save   | Rollback: delete the filesystem file       |

## 🔧 API Endpoints

All endpoints require Bearer API key authentication.

### File Upload (Multipart Form Data)

| Method | Path     | Description                              |
|--------|----------|------------------------------------------|
| `POST` | `/files/`| Upload a file (multipart form)            |

**Request:** `multipart/form-data`
- `file` (file): The file to upload
- `user_id` (int): Telegram user ID
- `tags` (string, optional): Comma-separated tags

**Response (201):**
```json
{"id": 1, "user_id": 123, "file_path": "123/report.pdf", "created_at": "2025-01-01T00:00:00"}
```

### File CRUD

| Method   | Path                    | Description                        |
|----------|-------------------------|------------------------------------|
| `GET`    | `/files/user/{user_id}` | Show all files for a user          |
| `DELETE` | `/files/{file_id}`      | Delete file from disk + DB (CASCADE)|

### Tag Operations

| Method   | Path                        | Description               |
|----------|-----------------------------|---------------------------|
| `GET`    | `/files/{file_id}/tags`     | Get tags for a file       |
| `POST`   | `/files/{file_id}/tags`     | Add tags to a file        |
| `DELETE` | `/files/{file_id}/tags`     | Remove tags from a file   |

## ⚙️ Implementation Notes

- **Upload:** `POST /files/` accepts `multipart/form-data` with `file`, `user_id`, optional `tags`
- **Tags** are case-normalized to lowercase
- **Duplicate tags** ignored via `UNIQUE` constraint + `ON CONFLICT DO NOTHING`
- **Transactions** for multi-step operations (file + tags)
- **Parameterized queries** via SQLModel/SQLAlchemy
- **Tables auto-created** on startup via `SQLModel.metadata.create_all`
- **Deletion order:** filesystem first, then database — if disk fails, DB is untouched

## 🚀 Setup

1. Set environment variables: `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`
2. Start the backend: tables are created automatically on startup
3. `files/` directory is created automatically on first upload
